Lesson 1: SSE 流式解析 — LLM API 的大门

📅 课程 1 / 6 ⏱ 12 分钟 📂 src/llm/llm.ts + sse.log
🎯 目标：理解 LLM 如何通过 SSE 协议"流式输出"响应，能读懂 sse.log 中的原始数据，并理解 ChatClient.chatStream() 的解析逻辑。

1. 问题：HTTP 如何传输"文字流"？

大模型生成一段文本不像数据库查询那样瞬间返回——它需要一个字一个字地"想"出来。如果等全部生成完毕再返回，用户要白等好几秒才能看到第一个字。

解决方案是 SSE（Server-Sent Events）：客户端发起一次 HTTP 请求后，服务端保持连接不断开，逐段推送数据。用户可以在大模型还没说完时就看到开头的内容。

💡 关键概念：SSE 不是 WebSocket。它是单向的（服务端→客户端），基于 HTTP 长连接，格式简单——每一行以 data: 开头。

2. 解剖 SSE 日志

打开项目中的 sse.log，你会看到大模型 GLM-Z1-9B 的一次完整流式输出。让我们逐行分析：

此外，docs/tutorials/sse-stream-parsing.md 提供了更详细的解析流程图和实现分析，可作为深入参考。

2.1 第一个 chunk：角色宣告

data: {"id":"019f3a68f625073d019a83c922d21d14","object":"chat.completion.chunk","created":1783391385,"model":"THUDM/GLM-Z1-9B-0414","choices":[{"index":0,"delta":{"content":"","role":"assistant"},"finish_reason":null}],"system_fingerprint":"","usage":{"prompt_tokens":15,"completion_tokens":0,"total_tokens":15}}

第一个 chunk 通常没有实际内容，只是声明 role: "assistant"，告诉客户端接下来要开始接收模型回复了。

2.2 推理过程（reasoning_content）

data: {"id":"019f3a68f625073d019a83c922d21d14","object":"chat.completion.chunk","created":1783391385,"model":"THUDM/GLM-Z1-9B-0414","choices":[{"index":0,"delta":{"content":null,"reasoning_content":"嗯","role":"assistant"},"finish_reason":null}],"system_fingerprint":"","usage":{"prompt_tokens":15,"completion_tokens":1,"total_tokens":16,"completion_tokens_details":{"reasoning_tokens":1}}}

注意 delta.reasoning_content 字段！这是思维链模型的推理过程，逐字输出。用户通常看不到这部分（在 TUI 中它被折叠显示），但它是理解模型"思考过程"的窗口。

2.3 最终回答（content）

当推理结束后，reasoning_content 变为 null，content 出现：

data: {"id":"019f3a68f625073d019a83c922d21d14","object":"chat.completion.chunk","created":1783391385,"model":"THUDM/GLM-Z1-9B-0414","choices":[{"index":0,"delta":{"content":"隐","role":"assistant"},"finish_reason":null}],"system_fingerprint":"","usage":{"prompt_tokens":15,"completion_tokens":300,"total_tokens":315,"completion_tokens_details":{"reasoning_tokens":221}}}

每个 chunk 只携带一个字符！这就是"流式"的体现——一个字一个字地推送。

2.4 结束标记

data: {"id":"..." ,"choices":[{"index":0,"delta":{"content":""},"finish_reason":"stop"}]}
...
data: [DONE]

最后两个信号：

  • finish_reason: "stop" — 模型告诉客户端"我说完了"
  • data: [DONE] — SSE 流的终止信号，之后不再有数据

3. 代码实现：ChatClient.chatStream()

打开 src/llm/llm.ts，核心方法是 chatStream()。让我们拆解它：

3.1 发起请求

llm.ts:70-78
const response = await fetch(`${model.baseUrl}/chat/completions`, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${model.apiKey}`,
  },
  body: JSON.stringify({ model, messages, stream: true, tools }),
});

关键点：stream: true——这告诉 API "我要流式响应，不要一次性返回"。

3.2 逐行解析 SSE 数据

llm.ts:85-126
const reader = response.body!.getReader();
const decoder = new TextDecoder();
let buffer = "";

// 逐块读取
while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split("\n");
  buffer = lines.pop() || "";  // 保留不完整的最后一行

  for (const line of lines) {
    if (!line.trim() || !line.startsWith("data: ")) continue;

    const data = line.slice(6);  // 去掉 "data: " 前缀
    if (data === "[DONE]") { /* 结束处理 */ }

    const parsed = JSON.parse(data);
    const delta = parsed.choices?.[0]?.delta;

    // 累加 content / reasoning_content / tool_calls
    if (delta?.content) content += delta.content;
    if (delta?.reasoning_content) reasoningContent += delta.reasoning_content;
  }
}
🔍 设计要点：
  • Buffer 技巧：TCP 传输可能把一个完整的 SSE 事件切碎，也可能一次携带多个事件。buffer 负责缓存不完整的尾部，下一次再拼接。
  • 逐 chunk 累加：content 和 reasoning_content 是逐字累加的，但 tool_calls 需要按 index 合并（因为一个工具调用的 name 和 arguments 可能分多个 chunk 到达）。
  • 超时保护：每读一个 chunk 有 30 秒超时，整体请求有 60 秒超时，防止连接卡死。

3.3 处理 Tool Call 增量

llm.ts:157-172
if (delta?.tool_calls) {
  for (const tc of delta.tool_calls) {
    const idx = tc.index ?? 0;
    if (!toolCalls[idx]) {
      toolCalls[idx] = { id: "", type: "function", function: { name: "", arguments: "" } };
    }
    if (tc.id) toolCalls[idx].id = tc.id;
    if (tc.function?.name) toolCalls[idx].function.name += tc.function.name;
    if (tc.function?.arguments) toolCalls[idx].function.arguments += tc.function.arguments;
  }
}

注意 tc.index——LLM 可能在一个响应中同时调用多个工具（"并行工具调用"）。每个工具调用通过 index 区分，系统必须按 index 分别累加。

5. 关键类型

打开 src/llm/types.ts，理解三个核心类型的关系：

types.ts:115-130
export interface LLMStreamChunk {          // 单个 chunk
  delta: {
    content?: string;
    reasoning_content?: string;
    tool_calls?: ToolCall[];
  };
  finish_reason?: "stop" | "tool_calls" | "length";
  usage?: TokenUsage;                      // 只有最后一个 chunk 有
  fullResponse?: LLMResponse;              // 只有最后一个 chunk 有（累加结果）
}
