📅 课程 4 / 6 ⏱ 14 分钟 📂 src/mcp/
🎯 目标：理解 Model Context Protocol (MCP) 如何在 babyAgent 中集成——从配置文件加载、传输层连接、工具发现，到适配为标准的 Tool 接口。

1. 为什么需要 MCP？

第 3 课的工具系统需要为每个工具手写代码实现。但如果想连接外部服务（浏览器、数据库、设计工具等），每个都手写显然不现实。

MCP (Model Context Protocol) 就是答案——它定义了一个标准化的工具服务器协议：

  • 统一接口：所有 MCP 服务器都实现 tools/list 和 tools/call 方法
  • 两种传输：本地进程（stdio）和远程 HTTP 服务（HTTP/SSE）
  • 即插即用：配置一个 mcp.json 文件，agent 自动发现工具
💡 项目当前配置了哪些 MCP 服务器？ 查看 src/mcp/config.ts —— 从 ~/.babyAgent/mcp.json 加载。
从 mcps/ 目录可以看到已经配置了：browser-use、context7、playwright、sequential-thinking 等服务器。

在 ~/.babyAgent/mcp.json 中的实际配置是这样：

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp"],
      "env": { "DEBUG": "true" }
    },
    "browser-use": {
      "url": "http://localhost:8123/sse"
    }
  }
}
🔍 设计要点：
  • Kebab-case 命名：服务器名必须小写字母+数字+连字符（如 my-server），提前发现命名错误
  • 互斥传输：command 和 url 只能二选一，不能同时设置
  • 空配置容错：文件不存在时返回空配置，不是报错

5. 工具适配：连接 MCP 与 Tool 接口

这是最巧妙的部分——src/mcp/tool-adapter.ts 把 MCP 工具转换成第 3 课学的 Tool 接口：

tool-adapter.ts:28-58
export function adaptMcpTool(serverName, mcpTool, transport): Tool {
  return {
    // ⭐ 工具名加前缀，防止不同服务器同名冲突
    name: `${serverName}_${mcpTool.name}`,

    description: mcpTool.description,

    // ⭐ 标准化 JSON Schema（处理 MCP 的灵活性）
    parameters: normalizeSchema(mcpTool.inputSchema),

    // ⭐ 执行时通过 transport.callTool 调用 MCP 服务器
    execute: async (params) => {
      const result = await transport.callTool(mcpTool.name, params);
      return {
        success: !result.isError,
        output: extractTextContent(result.content),
        error: result.isError ? extractTextContent(result.content) : undefined,
      };
    },
  };
}
⚡ 关键思维： 适配器（Adapter）模式！MCP 的工具定义 ≠ 项目的 Tool 接口，但通过 adaptMcpTool，MCP 工具看起来就像一个普通的 Tool。

适配后的工具名示例：

  • playwright_browser_navigate → 来自 playwright 服务器的 browser_navigate 工具
  • context7_resolve-library-id → 来自 context7 服务器的 resolve-library-id 工具
  • browser-use_navigate_page → 来自 browser-use 服务器的 navigate_page 工具

6. McpManager：把一切串起来

打开 src/mcp/mcp-manager.ts：

mcp-manager.ts:51-114
async loadAllTools(): Promise<Tool[]> {
  // 1. 加载并校验配置
  const { config, errors } = loadConfig(this.configPath);

  for (const [name, serverCfg] of Object.entries(config.servers)) {
    try {
      // 2. 根据配置创建对应传输层
      const t = await this.createTransport(name, serverCfg);

      // 3. 发现工具
      const mcpTools = await t.listTools();

      // 4. 适配为 Tool 接口
      const adapted = mcpTools.map(mcpTool => adaptMcpTool(name, mcpTool, t));

      allTools.push(...adapted);
    } catch (err) {
      // ⭐ 单个服务器失败不影响其他服务器
      console.error(`[MCP] ${name}: connection failed — ${err}`);
    }
  }

  return allTools;
}
🔍 设计要点：
  • 优雅降级：某个 MCP 服务器挂了不影响其他服务器，agent 照常启动
  • ServerStatus 追踪：记录每个服务器的连接状态和工具数量，供 UI 展示
  • 资源释放：dispose() 方法关闭所有传输连接，防止僵尸进程

7. 完整集成链路

现在是时候把 4 节课的知识串起来了：

 MCP 工具从配置到执行的完整链路

① 配置 ~/.babyAgent/mcp.json 定义服务器
② 加载 loadConfig() 校验并解析
③ 连接 createStdioTransport / createHttpTransport
④ 发现 client.listTools() → McpToolDef[]
⑤ 适配 adaptMcpTool() → Tool 接口实例（第 3 课）
⑥ 注册 加入 Agent 的 ToolRegistry
⑦ 调用 LLM 返回 tool_calls → agent 执行（第 2 课）→ transport.callTool()
⑧ 返回 提取文本内容，回到 ReAct 循环

8. 对比：自定义工具 vs MCP 工具

维度 自定义工具 (第 3 课) MCP 工具 (第 4 课)
实现方式 在项目内写代码 外部进程/服务，改配置即可
注册方式 代码中 toolRegistry.register() MCP 服务器启动后自动发现
适用范围 核心工具（bash、fs 等） 任何外部服务（浏览器、数据库等）
错误处理 try-catch 兜底 transport 层 + try-catch 双层保护
命名空间 短名称如 "bash" 带前缀如 "playwright_browser_navigate"
