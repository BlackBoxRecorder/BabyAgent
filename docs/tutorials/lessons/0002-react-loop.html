📅 课程 2 / 6 ⏱ 15 分钟 📂 src/agent.ts
🎯 目标：理解 ReAct（推理-执行）循环的完整流程——Agent 如何在 LLM 调用和工具执行之间反复切换，直到得到最终答案。

1. 什么是 ReAct？

ReAct = Reasoning + Acting，是一种让大模型交替进行"推理"和"行动"的模式。Agent 的核心循环是：

while (true) {
  思考 → 大模型输出（文字或工具调用）
  if (输出是最终回答) break;
  执行 → 调用外部工具，结果传回给大模型
  // 继续下一轮思考
}

为什么需要多轮？因为 HTTP 是无状态的——每次 LLM API 调用都是独立的。一轮调用结束后，需要通过 finish_reason 决定下一步：

  • finish_reason: "stop" → 模型说完了，结束循环
  • finish_reason: "tool_calls" → 模型要调用工具，把结果传回去继续
  • finish_reason: "length" → token 超限，异常结束

2. 源码解剖：runWithMessages()

打开 src/agent.ts，核心方法是 runWithMessages()。它是整个 agent 的"大脑"。

2.1 主循环

agent.ts:157-294
while (iterations < this.maxIterations) {
  iterations++;

  // 1. 流式调用 LLM
  for await (const chunk of this.llm.chatStream(messages, tools)) {
    yield { type: "chunk", chunk };
    if (chunk.fullResponse) fullResponse = chunk.fullResponse;
  }

  // 2. 根据 finish_reason 判断
  if (!fullResponse) { /* 无响应 → 失败 */ }
  else if (finish_reason === "length" / "content_filter" / ...) { /* 异常 → 失败 */ }
  else if (finish_reason === "stop") { /* 正常结束 → 返回结果 */ }

  // 3. 执行工具调用（finish_reason === "tool_calls"）
  // ...执行每个 tool，push 结果到 messages...
  // → 回到 while 开头，继续下一轮
}

注意：第 1 步和第 3 步之间没有 else 关键字——因为 finish_reason === "tool_calls" 时，代码会自然落到第 3 步执行工具，然后循环继续。这是 ReAct 的"行动"阶段。

🔍 设计要点：
  • messages 是局部变量：从 inputMessages 拷贝后，Agent 自己管理，不修改调用者的数组
  • streaming 贯穿始终：每一轮 LLM 调用都是流式的，通过 yield { type: "chunk" } 实时输出
  • fullResponse 只在最后一个 chunk 出现：这就是第 1 课中 LLMStreamChunk.fullResponse 的作用

2.3 工具调用的执行细节

当 finish_reason === "tool_calls" 时，agent 进入工具执行阶段：

agent.ts:212-293
// 先 push assistant 消息（包含 tool_calls）
messages.push({
  role: "assistant",
  content: fullResponse.content,    // 可能为 null
  tool_calls: fullResponse.tool_calls,
});

// 遍历每个工具调用
for (const toolCall of fullResponse.tool_calls ?? []) {
  const tool = this.toolRegistry.getTool(toolCall.function.name);

  // 安全执行：任何异常都被捕获，不会中断循环
  let result: ToolResult;
  try {
    const params = JSON.parse(toolCall.function.arguments);
    result = await tool.execute(params);
  } catch (err) {
    result = { success: false, output: "", error: String(err) };
  }

  yield { type: "tool_result", tool, params, result };

  // ⭐ 关键：将工具结果作为 tool 消息加入对话
  messages.push({
    role: "tool",
    tool_call_id: toolCall.id,
    content: JSON.stringify(result),
  });
}

// → while 循环继续，带着新消息再次调用 LLM
⚡ 消息结构演进：
一轮工具调用会在 messages 中插入 2 类新消息：
assistant (含 tool_calls) → tool (调用结果 1) → tool (调用结果 2) → ...
然后带着这些消息进入下一轮 LLM 调用——大模型"看到"了工具返回的结果，再决定下一步。

为什么工具错误也不中断循环？因为在 ReAct 模式下，大模型可以通过反馈自我修正。如果工具返回了错误，下一轮 LLM 调用时模型看到错误信息，可能会换一种方式调用，而不是让整个 agent 崩溃。

3. 理解 AgentStreamEvent

Agent 通过 yield 向外输出三种事件，供展示层（CLI/TUI）消费：

agent.ts:78-86
export type AgentStreamEvent =
  | { type: "chunk";       chunk: LLMStreamChunk }          ← SSE 流式 chunk
  | { type: "tool_result"; tool, params, result: ToolResult } ← 工具执行结果
  | { type: "done";        result: AgentResult }              ← agent 运行结束

这个设计的好处：

  • 解耦展示层：CLI 可以实时打印 chunk，TUI 可以更新状态面板，双方都不需要理解 LLM API 细节
  • 可测试性：测试框架可以收集所有事件，断言关键点的执行顺序
  • 可观察性：工具调用的参数和结果都暴露在事件中，方便调试和日志

4. 从第 1 课到第 2 课

回顾 SSE 解析中提到的 finish_reason——现在可以看到它在整个 ReAct 循环中的角色了：

📡 SSE 解析（第 1 课）→ 🧠 ReAct 循环（第 2 课）

SSE 解析的最终产出是 fullResponse（含 finish_reason 和 tool_calls）→
ReAct 循环消费 fullResponse，根据 finish_reason 做决策 →
需要执行工具时，调用 ToolRegistry 查找工具，执行后追加 tool 消息 →
带着新消息再次调用 LLM（回到 SSE 解析）→ ...
