3. AppFactory：一切组装的枢纽

打开 src/cli/app-factory.ts——这是所有模块的「接线板」：

app-factory.ts:53-161
export async function createApp() {
  // ① 模型配置
  const modelConfig = await loadModelConfig();
  const allModels = getAllModels(modelConfig);

  // ② 内置工具
  const bashTool = createBashToolAsTool(cwd);
  const fsTools  = createAllFsTools(cwd);

  // ③ MCP 工具（来自外部服务器）
  const mcpManager = new McpManager();
  const mcpTools   = await mcpManager.loadAllTools();

  // ④ 全部工具汇总
  const allTools = [bashTool, ...fsTools, ...mcpTools];

  // ⑤ Skill 发现
  const skillManager = new SkillManager();
  await skillManager.loadSkills();

  // ⑥ 系统提示词 + Skill 注入
  let systemPrompt = await loadSystemPrompt();
  systemPrompt += "\n\n" + skillManager.formatSkillsForSystemPrompt();

  // ⑦ 组装 Agent
  const agent = new Agent({
    llm:    new ChatClient(allModels, billingCalculator),
    tools:  allTools,
    systemPrompt,
  });

  // ⑧ 组装 Coordinator
  const coordinator = new ConversationCoordinator({
    agent,
    sessionManager: new SessionManager(),
  });

  return { coordinator, tools, mcpStatuses, models, ... };
}

这是你第一次看到整个系统的组装现场。注意 8 个步骤的顺序——每个组件依赖前序步骤产出的结果。

4. executeTurn()：一次对话的完整旅程

这是 Coordinator 最重要的方法——让我们追踪一条用户输入 "帮我查一下 CPU 温度" 的完整调用链：

 executeTurn("帮我查一下 CPU 温度") 全链路追踪

① 用户输入 → CLI/TUI 调用 coordinator.executeTurn("帮我查一下 CPU 温度")
② [无 session] → 自动创建会话 → yield { type: "session_created" }
③ 组装消息: [system prompt + skill 列表, ..., user: "帮我查一下 CPU 温度"]
④ 调用 agent.runWithMessages(messages) (第 2 课：ReAct 循环)
⑤ → LLM 返回 chunk 流 (第 1 课：SSE 解析)
⑥ → LLM 返回 tool_calls → agent 查 ToolRegistry（第 3 课）
⑦ → 可能是 MCP 工具（第 4 课），通过 transport.callTool() 执行
⑧ → 工具结果回到 LLM → LLM 生成最终回答 → finish_reason: "stop"
⑨ yield { type: "done", result } ← 所有流式事件向外吐出
⑩ 持久化: sessionManager.appendTurn() → .jsonl + .meta.json
🔍 设计亮点：
  • 流式事件转送：Agent 的 AgentStreamEvent（chunk / tool_result / done）被 Coordinator 原样 yield 出去，CLI/TUI 只需要消费
  • session_created 事件：UI 层收到后刷新 session 列表
  • 会话自动创建：用户第一条消息时自动创建 session，对用户完全透明

5. 错误处理的艺术

Agent 在复杂的工具调用中可能崩溃。Coordinator 有完善的保护机制：

coordinator.ts:219-271
// Agent 崩溃后的错误处理（重点！）
try {
  for await (const event of this.agent.runWithMessages(turnInput)) {
    ...
  }
} catch (agentErr) {
  // ① 保存错误现场到文件（防止 meta.json 孤儿）
  await this.sessionManager.appendTurn(this._sessionId, errorTurn);

  // ② 恢复消息历史到执行前状态
  this.agent.setMessages(
    this.agent.getMessages().slice(0, messagesBefore)
  );

  // ③ 向外吐出错误事件
  yield { type: "agent_error", error: errorMsg };
}

三点保障：

  • 防孤儿文件：session 已创建但 Agent 崩溃→保存错误 turn，避免只有 meta.json 没有 jsonl
  • 消息回滚：失败后消息历史恢复到执行前，下次对话从干净状态开始
  • 流式错误报告：agent_error 让 UI 层可以显示错误，而不是静默失败
