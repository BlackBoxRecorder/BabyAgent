2. Tool 接口：一切工具的契约

打开 src/tools/interface/index.ts——所有工具必须遵守这个接口：

index.ts:45-54
export interface Tool {
  name: string;                    // 唯一标识，LLM 通过 name 引用
  description: string;             // LLM 看的描述，决定何时使用
  parameters: JsonSchema;          // 参数定义（JSON Schema）
  execute: (params) => Promise<ToolResult>;  // 执行函数
}
💡 为什么叫"契约"？ 任何工具只要实现这 4 个字段，就能被注册表接纳、被 LLM 发现、被 Agent 调用。这就是策略模式——添加新工具不需要改 agent 的代码。

2.1 parameters: JSON Schema

工具的 parameters 使用 JSON Schema 格式，告诉 LLM：

  • 这个工具有哪些参数
  • 每个参数是什么类型（string / number / boolean / array）
  • 哪个参数是必填的
  • 每个参数的用途（description 字段）

以 bash 工具为例：

bash/tool.ts:27-40
parameters: {
  type: "object",
  properties: {
    command: {
      type: "string",
      description: "The bash command to execute",
    },
    timeout: {
      type: "number",
      description: "Optional timeout in seconds for the command",
    },
  },
  required: ["command"],           // command 是必填的
}

2.2 格式转换：给 LLM 看的样子

当 Tool 传给 LLM 时，需要包装成 API 认可的格式：

index.ts:78-98
export interface LLMFunctionDef {
  type: "function";
  function: {
    name: string;
    description: string;
    parameters: JsonSchema;
  };
}

// Tool → LLMFunctionDef 的转换：
export function formatToolForLLM(tool: Tool): LLMFunctionDef {
  return {
    type: "function",
    function: {
      name: tool.name,
      description: tool.description,
      parameters: tool.parameters,
    },
  };
}

这就是为什么第 1 课中 chatStream() 的 tools 参数类型是 LLMFunctionDef[]。转换后发给 API 的 JSON 大致是：

{
  "tools": [{
    "type": "function",
    "function": {
      "name": "bash",
      "description": "Execute bash commands in the terminal...",
      "parameters": {
        "type": "object",
        "properties": {
          "command": { "type": "string", "description": "..." },
          "timeout": { "type": "number", "description": "..." }
        },
        "required": ["command"]
      }
    }
  }]
}

3. ToolRegistry：工具的中枢

打开 src/tools/interface/registry.ts：

registry.ts:8-32
export class DefaultToolRegistry implements ToolRegistry {
  private tools: Map<string, Tool> = new Map();

  register(tool: Tool): void {
    if (this.tools.has(tool.name)) {
      throw new Error(`Tool "${tool.name}" is already registered`);
    }
    this.tools.set(tool.name, tool);
  }

  getTool(name: string): Tool | undefined {
    return this.tools.get(name);
  }

  getAllTools(): Tool[] { return Array.from(this.tools.values()); }

  getToolsForLLM(): LLMFunctionDef[] {
    return this.getAllTools().map(formatToolForLLM);
  }
}
🔍 设计要点：
  • Map 存储：O(1) 查找，适合 agent.ts 中频繁的 getTool(name)
  • 防重复注册：同名的工具不能注册两次，提前发现配置错误
  • 一键格式转换：getToolsForLLM() 批量转成 LLM 需要的格式，在 Agent 构造时调用一次

4. 具体工具：bash

打开 src/tools/bash/tool.ts：

bash/tool.ts:22-68
name: "bash",
description: "Execute bash commands in the terminal...",
parameters: {
  type: "object",
  properties: {
    command: { type: "string", description: "The bash command to execute" },
    timeout: { type: "number", description: "Optional timeout in seconds" },
  },
  required: ["command"],
},
async execute(params) {
  const result = await bashTool({ command: params.command, timeout: params.timeout });
  return {
    success: exitCode === 0,
    output: outputText,
    error: exitCode !== 0 ? `Exit code: ${exitCode}` : undefined,
    metadata: result.details,
  };
}

从 execute 的返回可以看出：

  • exit code === 0 → success
  • exit code !== 0 → 业务错误，把退出码当成 error 返回
  • 工具异常 → try-catch 兜底，error 字段保留消息

5. 具体工具：fs（Read/Write/Edit/Ls/Grep/Find）

打开 src/tools/fs/tool.ts，包含 6 个文件系统工具：

FS 工具一览

read — 读取文件内容
write — 写入/创建文件
edit — 精确替换文件中的文本（SearchReplace）
ls — 列出目录内容
grep — 搜索文件内容（基于 ripgrep）
find — 按 glob 模式查找文件（基于 fd）

每个 fs 工具都通过 wrapToolDefinition() 函数包装：

fs/tool.ts:84-111
function wrapToolDefinition(toolDef: ToolDefinition): Tool {
  return {
    name: toolDef.name,
    description: toolDef.description,
    parameters: toolDef.parameters,
    async execute(params) {
      const result = await toolDef.execute(params);
      const outputText = result.content
        .map((item) => item.text ?? item.data ?? "")
        .join("\n");
      return { success: true, output: outputText };
    },
  };
}

6. 完整链路回顾

现在我们可以看到从第 1 课到第 3 课的完整数据流：

 Agent 工具调用完整链路

① 初始化 Agent 构造时 → ToolRegistry.register(bash, read, write, ...)
② 格式转换 getToolsForLLM() → LLMFunctionDef[] → 放入 API 请求的 tools 参数
③ LLM 决策 大模型选择工具 bash({command: "ls -la"})，返回 tool_calls
④ 查找执行 agent.ts 中 getTool("bash") → tool.execute({command: "ls -la"})
⑤ 结果返回 ToolResult → 转为 tool 消息 → 追加到 messages → 下一轮 LLM 调用

7. 🧪 动手思考

如果你要添加一个新工具（比如 search_web），需要做哪些事？

  1. 创建一个新文件 src/tools/web/tool.ts（或类似的）
  2. 实现 Tool 接口：name、description、parameters、execute
  3. 在 Agent 构造时调用 toolRegistry.register(searchWebTool)
  4. 注册后，LLM 就能看到并调用它了！
⚡ 这就是策略模式的力量： 添加新工具 = 实现一个接口 + 注册一次。Agent 核心逻辑（ReAct 循环）不需要任何修改。
